# Security audit — Issues

**Verdict:** remediation-required  
**Date:** 2026-07-11  
**Scope:** Full application (`src/main/java`, `src/main/webui`, `application.properties`, keys, Flyway seed). Not production-deployed yet ([AGENTS.md](../AGENTS.md)); findings must be fixed before any shared or production deploy.

## Summary

Endpoint annotation hygiene is strong (119/119 `*Endpoint` classes use class-level `@DenyAll`; only six intentional `@PermitAll` auth paths). The dominant risk is **missing project-membership authorization on ticket and several project-adjacent APIs** — any authenticated user can read and mutate tickets outside projects they belong to. The repo also ships a **development-only JWT keypair** and a **global password salt / known default password hash** (acceptable for local `%dev`/`%test`; must not be reused for shared/production). Injection (JPQL/SQL/LDAP filter escape) looks solid; XSS risk is residual via rich-text `[innerHTML]` plus tokens in `localStorage`.

## Coverage

| Area | Reviewed |
|------|----------|
| Endpoints | All `*Endpoint` — `@DenyAll` / `@RolesAllowed` / `@PermitAll` |
| Auth | Login, register, recovery, refresh, capabilities, JWT issuer, PAT/SA, LOCAL/LDAP/ENDPOINT |
| AuthZ | `ProjectAccessService`, `TicketService`, import, dashboard, phase, notifications, users |
| Injection | JPQL/native params, LDAP filter escape |
| Secrets | `application.properties`, `privateKey.pem` / `publicKey.pem`, mailer `%dev` |
| Frontend | Guards, token storage, `[innerHTML]`, login redirect, CORS |
| Uploads | CSV import size/row limits, importId ownership |
| Tests | Skimmed only where they would weaken prod (no intentional security bypass found beyond normal `@QuarkusTest`) |

**Intentional skips:** dependency CVE enumeration (no SCA run); full LDAP live testing; production Docker/K8s manifests beyond properties.

## Positive controls

- Class-level `@DenyAll` on all endpoints; public surface limited to `/auth/*` login/register/recovery/refresh/capabilities.
- Refresh token rotation + revoke-all on password change/reset.
- LDAP filter values escaped (`JndiLdapDirectoryClient.escapeFilterValue`).
- Parameterized JPQL / native queries; no string-concat query builders found.
- PAT/SAT secrets hashed (SHA-256); PAT revoke scoped to owner; SA create uses `requireManage`.
- Strong password policy (`@StrongPassword`); recovery silent no-op for unknown emails.
- CSV upload: 5 MB / 500 row caps; filename is metadata only (no path write).
- `ProjectAccessService.requireView` / `requireManage` used correctly on members, custom fields, backlog, burndown, ticket links, service accounts.

## Findings

### SEC1 — Ticket IDOR: no project membership on ticket APIs (critical)

**Location:** `TicketService` (`listAll`, `findById`, `create`, `update`, `search`, `requireTicketForView`); `FindTicketByIdEndpoint`; `ListTicketsEndpoint`; ticket search package; create/update/move/comment paths.

**Issue:** `requireTicket` / `requireTicketForView` only load the ticket (and optionally allow soft-deleted for admin/PM). They never call `projectAccessService.requireView`. `listAll` / `search` return tickets across all projects. `create` does not check that the author is a member of `request.projectId()`. Contrast: `findByProjectId` and `TicketLinkService` correctly call `requireView`.

**Impact:** Any authenticated `user` who can log in can list, read, edit, move, comment on, and create tickets in projects they are not allocated to. Breaks project isolation (core multi-tenant boundary).

**Evidence:**

```139:141:src/main/java/dev/vepo/issues/ticket/TicketService.java
    public TicketResponse findById(long id) {
        return toResponse(requireTicket(id));
    }
```

```505:511:src/main/java/dev/vepo/issues/ticket/TicketService.java
    private Ticket requireTicketForView(long id, String username) {
        var user = requireUserByUsername(username);
        if (canViewDeletedTicket(user)) {
            return repository.findByIdIncludingDeleted(id)
                             .orElseThrow(() -> ticketNotFound(id));
        }
        return requireTicket(id);
    }
```

```38:39:src/main/java/dev/vepo/issues/ticket/find/FindTicketByIdEndpoint.java
    public TicketResponse findById(@PathParam("id") Long id) {
        return ticketService.findById(id);
```

**Fix task SEC1**

**Goal:** Enforce project view (and manage where needed) on every ticket read/write path; scope list/search to the caller’s project membership (or admin).

**Do not:** weaken tests by removing membership assertions; skip create/import paths.

**Steps for the developer:**

1. Add `projectAccessService.requireView(ticket.getProject().getId(), username)` inside a shared helper used by all ticket loaders (`requireTicketForActor`, etc.).
2. Pass `SecurityContext` / username into `findById` and all mutating methods that currently omit it.
3. Change `listAll` / simple search / query language execution to filter by `projectAccessService.projectScopeIds(user)` (admin = all).
4. On `create`, call `requireView` (or membership) for `request.projectId()` before persist.
5. Align ticket import ticket-creation path with the same checks.

**Code pattern to follow:** `TicketService.findByProjectId` (`requireView` at line 133); `TicketLinkService` membership checks; `BurndownService` (`requireView`).

**Verify:** `@QuarkusTest` — member of project A can GET A tickets; user only in B gets 403/404 for A’s ticket id/identifier; `listAll`/search never returns foreign tickets; create into foreign project fails. Run `TicketEndpointTest` + search tests.

**Process:** extend `feature/ticket-management.md` (or new `feature/ticket-authorization.md`); architecture note on membership gate; obtain task approval before `src/**` edits.

### SEC2 — JWT signing private key committed for development only (medium)

**Location:** `src/main/resources/privateKey.pem`; `smallrye.jwt.sign.key.location=privateKey.pem` in `application.properties`; verify key `META-INF/resources/publicKey.pem`.

**Clarification (product / ops intent):** This keypair was **generated for the development environment only** (`%dev` / local Quarkus, and tests that load the same classpath resources). It is **not** a production secret and must **never** be used to sign or verify JWTs on shared staging, demo, or production hosts.

**Issue:** The committed key is intentional for local development convenience. Residual risk is **accidental reuse**: default `application.properties` points at `privateKey.pem` without a `%prod` override, so a careless deploy that does not supply environment-specific keys would trust a well-known repo keypair. Anyone with repo access can forge tokens **only against deployments that still use this development key**.

**Impact:** **Acceptable** for isolated local `%dev`/`%test`. **High** if the same key is left as the effective signing/verify material outside development. Not a stop-ship for local work; **must** be overridden before any non-dev deploy.

**Evidence:** `privateKey.pem` / `publicKey.pem` in resources; `application.properties:60` (default location, not `%dev`-scoped today); comments at `:32–35` already discuss prod issuer/JWKS overrides but do not yet state explicitly that the bundled PEM pair is **development-only**.

**Fix task SEC2**

**Goal:** Make the **development-only** nature of the bundled JWT keypair explicit in config and docs; ensure `%prod` (and any shared staging) **cannot** silently use the repo PEMs.

**Do not:** treat the committed key as a production incident requiring rotation of “leaked prod secrets” — it was never generated for production. Do not leave non-dev deploys on the default `privateKey.pem` path.

**Steps for the developer:**

1. Document next to the key paths (and in `SECURITY.md` / README): **“Bundled `privateKey.pem` / `publicKey.pem` are for development and test only.”**
2. Scope or comment default sign/verify locations so `%prod` **requires** env/file overrides (`smallrye.jwt.sign.key.location`, `mp.jwt.verify.publickey.location` or JWKS) — fail startup if missing in prod.
3. Keep the committed pair for `%dev`/`%test` local convenience (acceptable).
4. Ops checklist: any shared/demo/prod environment must generate its **own** keypair (or use a secrets manager); never copy the repo PEMs.

**Code pattern to follow:** Existing prod override comments at `application.properties:32–35`; mirror for signing key with an explicit **development-only** note on the bundled files.

**Verify:** `%dev` still boots with repo keys; `%prod` (or Docker prod image) refuses to start without external keys, or starts only with non-repo keys.

**Process:** Docs/config under `feature/authentication.md` or ops notes; approval if behaviour of prod profile changes.

### SEC3 — Global PBKDF2 salt and known default password hash (high)

**Location:** `PasswordEncoder` + `application.properties` `password.salt`, `password.default` with comment `## qwas1234`; Flyway/dev seed use the same hash.

**Issue:** All local passwords share one salt (rainbow / batch cracking easier). Default password plaintext is documented in-repo next to the hash. Seed users all share that password.

**Impact:** Offline cracking of stolen hashes is cheaper; any exposed DB or hash dump with default users is immediately usable with `qwas1234` (or whatever default is documented).

**Evidence:** `PasswordEncoder.java:33–43,57–60`; `application.properties:40–42`.

**Fix task SEC3**

**Goal:** Per-user random salt stored with the hash; remove plaintext default password from committed config; force unique strong passwords in non-dev or rotate on first login.

**Do not:** keep a shared salt “for simplicity”; leave `## qwas1234` in production-bound files.

**Steps for the developer:**

1. Change hash format to include salt (e.g. `{iterations}${saltB64}${hashB64}` or dedicated columns).
2. Migrate existing hashes (re-hash on next login or one-time reset for LOCAL users).
3. Use `MessageDigest.isEqual` (or equivalent) for constant-time compare of digests (also addresses SEC9).
4. Scope `password.default` to `%dev` / `%test` only; document that production must not use it.
5. Ensure register / admin create paths always use the new encoder.

**Code pattern to follow:** `ApiTokenHasher` (per-secret random + hash at rest) for “unique secret material” mindset.

**Verify:** Two users with same password have different stored hashes; unit test `matches`; login still works after migration.

**Process:** `feature/authentication.md`; schema change in `V1.0.0` (pre-prod) per flyway rule.

### SEC4 — Stored HTML rendered via `[innerHTML]` (rich text) + tokens in localStorage (high)

**Location:** Angular templates (`ticket-view`, `kanban`, `home`, `project-hub`, comments, custom fields); `rich-text-editor.component.ts` writes `innerHTML` from editor; `AuthService` stores JWT + refresh in `localStorage`.

**Issue:** User-controlled HTML (description, comments, project description, Text CFs) is bound with `[innerHTML]`. Angular sanitizes many script vectors, but rich-text + `contenteditable` still expands XSS/CSS-injection surface. XSS in the SPA can steal `localStorage` JWTs.

**Impact:** Cross-user XSS → session/token theft → full API access as victim (amplified by SEC1).

**Evidence:** `ticket-view.component.html` `[innerHTML]="ticket.description"` / `comment.content`; `auth.service.ts` `jwt_token` / `jwt_refresh_token` keys.

**Fix task SEC4**

**Goal:** Sanitize rich-text on **write** (server) and/or strict allowlist on **read**; reduce token theft impact (httpOnly cookie or memory + short TTL — product decision).

**Do not:** use `bypassSecurityTrustHtml`; store raw untrusted HTML forever without a sanitizer.

**Steps for the developer:**

1. Add server-side HTML sanitizer (OWASP Java HTML Sanitizer or equivalent) on ticket description, comments, project description, Text CF values before persist.
2. Keep Angular `[innerHTML]` only for already-sanitized HTML; document allowlist (tags/attrs).
3. Optionally move access token to memory-only and refresh via httpOnly cookie (CSRF then matters — design carefully), or accept localStorage but prioritize write-sanitize.
4. Add regression tests: payload with `<script>`, `onerror=`, `javascript:` URLs stripped.

**Code pattern to follow:** Treat like validation on `CreateTicketRequest` / comment create services — sanitize in `*Service` before repository save.

**Verify:** Malicious HTML stored via API does not execute when another user opens ticket detail; tokens not exfiltrated in a manual XSS check.

**Process:** extend `feature/rich-text-editor.md`; domain note on trusted HTML; approval before implementation.

### SEC5 — Notification mark-as-read IDOR (high)

**Location:** `UpdateNotificationReadEndpoint` → `NotificationService.markAsRead` — no receiver/username check.

**Issue:** Any authenticated user who knows (or guesses) a notification id can mark another user’s notification read/unread. List/count correctly filter by username; mark-all is scoped; single-id update is not.

**Impact:** Integrity/privacy of notification state; possible probing of notification id existence.

**Evidence:**

```30:35:src/main/java/dev/vepo/issues/notifications/NotificationService.java
    public UserNotificationEvent markAsRead(long notificationId, UpdateNotificationStatusReadRequest request) {
        var notification = notificationRepository.findById(notificationId)
                                                 .orElseThrow(NotFoundException::new);
        notification.setRead(request.read());
        return UserNotificationEvent.load(notificationRepository.save(notification));
    }
```

Endpoint does not pass `SecurityContext` username (`UpdateNotificationReadEndpoint.java:41–43`).

**Fix task SEC5**

**Goal:** Only the notification owner (or admin, if product allows) may update read state.

**Do not:** return different errors that enable user enumeration beyond existing list behaviour.

**Steps for the developer:**

1. Change `markAsRead` to accept `username` and load via `findByIdAndUsername` (or check `notification.getUser()`).
2. Pass principal from endpoint `SecurityContext`.
3. Return 404 when not owner (prefer over 403 to avoid id probing, consistent with list).

**Code pattern to follow:** `markAllAsRead(username)` / `list(username, …)` already scoped by username.

**Verify:** User A cannot mark User B’s notification; owner can. Extend notification endpoint tests.

**Process:** `feature/notifications.md` changelog; approve tasks before code.

### SEC6 — CSV import lacks membership and import ownership checks (high)

**Location:** `TicketImportService.upload` / `requireImport`; project and global upload endpoints.

**Issue:** Project-scoped upload does not call `requireView`/`requireManage` on `projectId`. `requireImport` only matches import id (+ optional project id) — **any** authenticated user who learns `importId` can map/preview/execute another user’s import and create tickets (compounded by SEC1).

**Impact:** Cross-user import takeover; create tickets into projects without allocation.

**Evidence:** `upload` sets author but later ops use `requireImport` without author check (`TicketImportService.java:72–99`, `:433–439`).

**Fix task SEC6**

**Goal:** Require project view (or manage) on project import; bind subsequent importId operations to the uploading user (or admin).

**Do not:** allow import execution without re-checking project membership at create time.

**Steps for the developer:**

1. On project upload: `projectAccessService.requireView(projectId, username)` (or stricter).
2. Change `requireImport` to `findByIdAndAuthor` (and project when scoped).
3. Ensure row execution reuses SEC1 membership on ticket create.
4. Add tests for foreign importId → 404 and non-member project upload → 403.

**Code pattern to follow:** `ProjectAccessService.requireView`; author binding similar to saved-query owner checks.

**Verify:** Member can import; non-member cannot; user B cannot continue user A’s importId.

**Process:** `feature/ticket-import.md`; approve before code.

### SEC7 — Dashboard / phase / version APIs missing `requireView` / `requireManage` (high)

**Location:** `DashboardService` (layout + KPI/pie/table — project exists only); `PhaseService` / version services (no `ProjectAccessService`); contrast `BurndownService` which calls `requireView`.

**Issue:** Any authenticated user can read dashboard aggregates and phase/version data for any `projectId` if they know the id. Phase mutate endpoints rely on role annotations (admin/PM) without owner/membership checks — a global PM may mutate phases on projects they do not own (product may intend this; document or tighten).

**Impact:** Cross-project information disclosure (ticket counts, priorities, recent tickets). Possible unauthorized planning changes depending on intended PM semantics.

**Evidence:** `DashboardService.getLayout` / `loadPieData` call `requireProject` only; burndown uses `projectAccessService.requireView`.

**Fix task SEC7**

**Goal:** Align dashboard/phase/version **read** with `requireView`; align **mutate** with `requireManage` (owner/admin) unless product explicitly allows any PM.

**Do not:** leave dashboard as the odd one out vs burndown.

**Steps for the developer:**

1. Inject `ProjectAccessService` into `DashboardService` and phase/version services.
2. Call `requireView` on all reads; `requireManage` on create/update/activate/complete (confirm with product for PM-global vs owner-only).
3. Mirror `feature/project-dashboard.md` AQ2 (`requireView`) — already planned for dashboard hardening.

**Code pattern to follow:** `BurndownService` lines using `requireView`.

**Verify:** Non-member gets 403 on dashboard/phase list for foreign project; member succeeds. Endpoint tests.

**Process:** `feature/project-dashboard.md` T-tasks / `feature/phase-management.md`; approve before code.

### SEC8 — No rate limiting on auth endpoints (medium)

**Location:** `POST /auth/login`, `/auth/register`, `/auth/recovery`, `/auth/refresh` — no application-level throttle found.

**Issue:** Online password spraying, recovery email flooding, and refresh token stuffing are unconstrained at the app layer.

**Impact:** Account takeover attempts and mailer abuse against LOCAL deployments.

**Fix task SEC8**

**Goal:** Rate-limit auth endpoints by IP and/or account key; lockout or backoff on repeated failures.

**Do not:** log passwords; return whether email exists on recovery (keep silent no-op).

**Steps for the developer:**

1. Add Quarkus/SmallRye rate limiting or a simple bucket (IP + email) on login/register/recovery/refresh.
2. Optional progressive delay / captcha for public internet deploy.
3. Document reverse-proxy limits as defense in depth.

**Verify:** N failed logins from one IP within window → 429; legitimate login still works.

**Process:** `feature/authentication.md`; approve before code.

### SEC9 — Non-constant-time password comparison (medium)

**Location:** `PasswordEncoder.matches` uses `String.compareTo` on Base64 hashes.

**Issue:** Timing side channel on hash string compare (lower practical risk than SEC1/SEC3 but easy to fix with SEC3).

**Fix task SEC9**

**Goal:** Compare digests with `MessageDigest.isEqual` on decoded bytes.

**Steps:** Implement as part of SEC3 encoder redesign; add unit test that matches still works.

**Process:** Same as SEC3.

### SEC10 — LDAP group mapping uses substring `contains` (medium)

**Location:** `LdapCredentialAuthenticator.matchesGroup` — equality **or** `directoryGroup.contains(mappedGroup)`.

**Issue:** A group DN/CN that merely contains a configured substring can grant ADMIN/PROJECT_MANAGER.

**Impact:** Privilege escalation when LDAP group names are attacker-influenced or loosely mapped.

**Evidence:** `LdapCredentialAuthenticator.java:73–78`.

**Fix task SEC10**

**Goal:** Exact match only (or normalized DN equality), never substring.

**Steps:** Remove `contains` branch; document config must use full group CN/DN; add unit tests for false-positive substrings.

**Process:** `feature/authentication.md` LDAP notes; approve before code.

### SEC11 — Swagger UI always included (medium)

**Location:** `quarkus.swagger-ui.always-include=true`; path `/openapi`.

**Issue:** Full API surface documentation exposed on every profile, including deployments that forget to disable it.

**Impact:** Easier recon for attackers; not a direct vuln but increases attack surface.

**Fix task SEC11**

**Goal:** Enable Swagger only for `%dev` / `%test`; disable in `%prod` by default.

**Steps:** Profile-gate `always-include`; document how to enable for internal staging.

**Process:** Ops / `feature/authentication` or platform docs; low process weight if config-only.

### SEC12 — User admin SPA routes lack `roleGuard` (medium)

**Location:** `app.routes.ts` `/users`, `/users/new`, `/users/:userId` — `authGuard` only. Mutating APIs correctly `@RolesAllowed(ADMIN)`.

**Issue:** UI-only gap; any authenticated user can load the users UI and list/find APIs (search/find allow USER+). Create/update/delete still blocked by API.

**Impact:** User directory disclosure and admin UX confusion; not full privilege escalation if API holds.

**Fix task SEC12**

**Goal:** Add `roleGuard(['admin'])` to user routes; optionally tighten find/search to admin-only if product agrees.

**Code pattern:** `/categories` already uses `roleGuard(['admin'])`.

**Verify:** Non-admin redirected from `/users`; admin can open. API tests unchanged for mutate.

**Process:** `feature/user-management.md`; approve before code.

### SEC14 — Refresh and password-reset tokens stored as UUID plaintext (low)

**Location:** `RefreshToken` / `PasswordResetToken` use `UUID.randomUUID()` stored in clear in DB.

**Issue:** DB leak yields usable session refresh / reset tokens until expiry/revoke. Entropy of UUID is acceptable for guessing, but hashing at rest is better practice (as done for API tokens).

**Fix task SEC14**

**Goal:** Store only hash of refresh/reset tokens; compare on lookup (same pattern as `ApiTokenHasher`).

**Process:** `feature/authentication.md`; schema + service changes with approval.

### SEC15 — Create/update project role annotation vs admin manage (info / low)

**Location:** `UpdateProjectEndpoint` / `CreateProjectEndpoint` `@RolesAllowed(PROJECT_MANAGER)` only; `ProjectAccessService.canManage` includes admin.

**Issue:** Admin without `project-manager` sees hub **Editar** but API returns 403. Availability/consistency issue more than escalation.

**Fix task SEC15**

**Goal:** Align `@RolesAllowed` with `canManage` (include ADMIN) **or** hide UI for admin-without-PM.

**Process:** `feature/project-administration.md`.

## Recommended next steps

1. **Prioritize SEC1** (ticket / project visibility) before any shared staging deploy — membership IDOR dominates risk.
2. **SEC2:** treat bundled JWT PEMs as **development-only**; before shared/prod deploy, supply environment-specific keys and document that the repo pair must not be copied (hygiene, not “stolen prod key” rotation).
3. Bundle SEC3+SEC9, SEC5, SEC6, SEC7 as the next authz/hardening wave (SEC7 overlaps planned dashboard hardening).
4. For each accepted Fix task: feature analysis / changelog → architecture if needed → task break → **explicit task ID approval** → TDD ([development-process.mdc](../.cursor/rules/development-process.mdc)). This audit does **not** implement fixes.
5. After remediations: re-run `/review_security` (or scoped package audit) and update [SECURITY.md](../SECURITY.md) with any new operational requirements (dev-only JWT note, key env vars, Swagger off in prod).
