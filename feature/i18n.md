# UI internationalization (i18n)

**Feature version:** 1  
**Status:** tasks-ready  
**Requested:** 2026-07-11

## Summary

Ship **UI internationalization** so Issues presents chrome, labels, and **system labels** in **English** or **Portuguese**, with **`pt` as the Angular source locale**. Language is chosen in **account settings** and stored on the **user** profile; unauthenticated visits follow the **browser** `Accept-Language`. Locale URLs use path prefixes **`/pt/`** and **`/en/`**. Preference is **seeded on register / first provisioning**. API errors and emails stay as today. User-authored content is never auto-translated.

Backlog: [docs/backlog.md](../docs/backlog.md). Deferred earlier from [ui-design-system.md](ui-design-system.md) Q3.

## Architecture

| Area | Design |
|------|--------|
| Bounded contexts | Identity & access owns **locale preference**; UI strings are frontend-only |
| Packages / layers | `User.uiLocale`; `AuthenticationService` register/updateProfile/me/provision; endpoints unchanged structurally |
| API | `AuthResponse.locale`; `UpdateProfileRequest.locale` (allow-list `pt`\|`en`); register/provision seeds locale from request body locale and/or `Accept-Language` (map ŌåÆ `pt`\|`en`, else `pt`) |
| Schema | `tb_users.ui_locale VARCHAR(8) NOT NULL DEFAULT 'pt'` in `V1.0.0`; seed personas `pt` |
| Cross-context | None |
| Frontend | Compile-time i18n; `baseHref` `/pt/` and `/en/`; shared `system-labels.ts`; account settings select; reload to other prefix on save |
| Serve | Root `/` ŌåÆ 302 to `/pt/` or `/en/` from `Accept-Language`; static assets under each prefix; authenticated mismatch ŌåÆ redirect to preference prefix |
| Dev (**AQ3**) | `quarkus:dev` / Quinoa serves **one** locale (default `pt`); toggle via npm script / env (e.g. `ISSUES_UI_LOCALE=en`); CI runs production build for **both** locales |
| Tests | Endpoint + Angular + dual-locale build smoke in CI |
